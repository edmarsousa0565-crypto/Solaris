🔴 Semana 1 — Impacto Máximo

Imagens — Economia de 476 KiB

O problema mais grave. Estás a servir imagens sem otimização — provavelmente PNG/JPG grandes sem compressão, sem WebP, sem lazy loading e sem tamanhos responsivos. O Next.js tem o componente Image que resolve tudo automaticamente.

476 KiB de poupança — isto sozinho pode reduzir o LCP de 8.4s para ~4s. É a correção de maior impacto.

Correção — Substituir todas as <img> por next/image

❌ Antes — Errado
// Imagem normal — sem otimização
<img
  src="/hero.jpg"
  alt="Coleção Verão"
  style={{ width: '100%' }}
/>
✓ Depois — Correto
import Image from 'next/image'

// Otimizado — WebP automático
<Image
  src="/hero.jpg"
  alt="Coleção Verão SOLARIS"
  width={1200}
  height={800}
  priority
  sizes="100vw"
/>

Para imagens de produto (lazy load)

components/ProductCard.tsx
import Image from 'next/image'

export function ProductCard({ product }) {
  return (
    <div className="product-card">
      <Image
        src={product.image}
        alt={product.name}
        width={600}
        height={800}
        // SEM priority = lazy load automático
        sizes="(max-width: 768px) 100vw, 50vw"
        className="product-img"
      />
    </div>
  )
}

Para imagens externas (CJdropshipping / Printful)

next.config.js
// Adicionar domínios externos permitidos
const nextConfig = {

🔴 Semana 1

JavaScript Não Usado — 97 KiB

Tens 97 KiB de JavaScript carregado na página inicial que não é usado. Isto atrasa o FCP e bloqueia o thread principal. A solução é usar dynamic imports para componentes pesados que não são necessários no carregamento inicial.

Dynamic imports para componentes pesados

❌ Antes — Carrega tudo de imediato
// Carrega na página inicial
// mesmo que o utilizador nunca abra
import { CartDrawer } from './CartDrawer'
import { SearchModal } from './SearchModal'
import { SizeGuide } from './SizeGuide'
✓ Depois — Carrega só quando precisa
import dynamic from 'next/dynamic'

// Carrega só quando o utilizador abre
const CartDrawer = dynamic(
  () => import('./CartDrawer'),
  { loading: () => <Spinner /> }
)

const SearchModal = dynamic(
  () => import('./SearchModal')
)

const SizeGuide = dynamic(
  () => import('./SizeGuide')
)

Verificar o bundle — ver o que está a pesar

Terminal
# Instalar analisador de bundle
npm install @next/bundle-analyzer

# next.config.js
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
})

# Correr análise
ANALYZE=true npm run build
Dica: Verifica se tens bibliotecas de animação (GSAP, Framer Motion) a carregar na homepage. Estas são frequentemente a causa de JS não usado — usar dynamic import com ssr: false resolve.

🔴 Semana 1 — LCP

Preload da Imagem Hero

O browser está a descobrir a imagem hero tarde demais — só depois de processar todo o CSS e JS. Preloading diz ao browser para começar a descarregar a imagem imediatamente, antes de processar qualquer outra coisa.

❌ Antes — Descoberto tarde
// Hero sem priority
<Image
  src="/hero.jpg"
  alt="Hero"
  width={1200}
  height={800}
/>
✓ Depois — Preload imediato
// priority = preload automático
<Image
  src="/hero.jpg"
  alt="Coleção Verão SOLARIS"
  width={1200}
  height={800}
  priority
  sizes="100vw"
  quality={85}
/>

O atributo priority no next/image adiciona automaticamente rel="preload" no head e desativa o lazy loading. Usar apenas na imagem above-the-fold (hero).

Solução — Criar ficheiro correto

❌ Atual — Com erros
# Provavelmente inválido ou ausente
# 23 erros encontrados pelo Google

User-agent: *
Allow: /

# Bloquear páginas de API e privadas

Criar sitemap dinâmico em Next.js

app/sitemap.ts

  return [
    {
      url: 'https://solarisstore.pt',
      lastModified: new Date(),
      changeFrequency: 'weekly',
      priority: 1,
    },
    {
      url: 'https://solarisstore.pt/shop',
      lastModified: new Date(),
      changeFrequency: 'daily',
      priority: 0.9,

  ]

🟡 Semana 2 — Acessibilidade

Botões sem Nome Acessível

Botões de ícone (carrinho, pesquisa, menu hamburger, fechar) não têm texto nem aria-label. O Google e leitores de ecrã não conseguem interpretar o que fazem, o que penaliza o score de acessibilidade.

❌ Antes — Sem nome
<button>
  <CartIcon />
</button>

<button>
  <SearchIcon />
</button>

<button>
  <MenuIcon />
</button>
✓ Depois — Com aria-label
<button aria-label="Abrir carrinho">
  <CartIcon />
</button>

<button aria-label="Pesquisar produtos">
  <SearchIcon />
</button>

<button aria-label="Abrir menu">
  <MenuIcon />
</button>

🟡 Semana 2 — Acessibilidade

Contraste Insuficiente

Texto claro sobre a imagem hero não tem contraste suficiente (mínimo 4.5:1 exigido). Isto afeta legibilidade em mobile sob luz solar direta — muito comum em Portugal no verão.

❌ Antes — Texto sem overlay
/* Texto direto sobre imagem */
.hero {
  background-image: url('/hero.jpg');
}

.hero-text {
  color: white;
  /* sem overlay = contraste insuficiente */
}
✓ Depois — Com overlay escuro
/* Overlay garante contraste */
.hero {
  position: relative;
}

.hero::before {
  content: '';
  position: absolute;
  inset: 0;
  background: rgba(0,0,0,0.45);
  z-index: 1;
}

.hero-text {
  position: relative;
  z-index: 2;
  color: white;
  text-shadow: 0 2px 8px rgba(0,0,0,0.5);
}

🟡 Semana 2 — SEO + Acessibilidade

Hierarquia de Headings

Os headings têm de seguir ordem sequencial: h1 → h2 → h3. Saltar níveis (ex: h1 → h3) confunde os leitores de ecrã e prejudica o SEO.

❌ Antes — Ordem incorreta
<h1>SOLARIS</h1>
<!-- Salta h2! -->
<h3>Coleção Verão</h3>
<h4>Vestidos</h4>
<h4>Conjuntos</h4>
✓ Depois — Ordem correta
<h1>SOLARIS</h1>
<!-- h2 sempre a seguir a h1 -->
<h2>Coleção Verão</h2>
<h3>Vestidos</h3>
<h3>Conjuntos</h3>
Regra: Cada página deve ter exatamente 1 h1 com a palavra-chave principal (ex: "Vestidos Verão Portugal"). Os h2 são secções principais, h3 são subsecções.
